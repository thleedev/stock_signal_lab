# 투자시황 스코어링 투명화 + 체크리스트 판정 시스템

> 2026-03-29 | 기존 종목추천 스코어링에 근거 레이어를 추가하고, 조건 기반 체크리스트 페이지를 신설한다.

---

## 1. 배경 및 목표

### 문제
- 종목추천의 6차원 점수(신호·추세·밸류·수급·이익모멘텀·리스크)가 **왜 그 값인지 근거가 없음**
- 정규화 기준이 제각각 (추세 0~65, 밸류 0~25, 수급 -10~45) → 비직관적
- 각 스코어 함수가 boolean만 반환 (`golden_cross: true`) → 실제 수치 근거 부재

### 목표
1. **접근법 A**: 기존 스코어링에 근거 데이터를 추가하여 "왜 이 점수인지" 투명하게 표시
2. **접근법 C**: 조건 기반 체크리스트 판정 시스템을 새 페이지로 신설, 병행 운영

### 범위
- 시가총액 티어별 차등 가중치 체계는 유지
- 기존 스코어링 히스토리와의 호환은 고려하지 않음 (정규화 체계 변경)

## 2. 접근법 A — 근거 레이어 + 정규화 통일

### 2.1 ScoreReason 타입

```typescript
// types/score-reason.ts
interface ScoreReason {
  label: string;      // "골든크로스"
  points: number;     // +8 (정규화 후 기여 점수)
  detail: string;     // "5일선 12,340 > 20일선 12,100 (3/28 상향돌파)"
  met: boolean;       // 조건 충족 여부
}
```

모든 스코어 모듈의 Result 인터페이스에 `reasons: ScoreReason[]` 필드를 추가한다.

### 2.2 정규화 통일

모든 모듈이 내부에서 **0~100** 스케일로 정규화하여 반환한다.

| 모듈 | 현재 범위 | 변경 후 | 방법 |
|------|----------|---------|------|
| signal-score | 0~30 | 0~100 | `(raw / 30) * 100` |
| technical-score | 0~65 | 0~100 | `(raw / 65) * 100` |
| valuation-score | 0~25 | 0~100 | `(raw / 25) * 100` |
| supply-score | -10~45 | 0~100 | `((raw + 10) / 55) * 100` |
| earnings-momentum-score | 0~100 | 0~100 | 이미 통일 |
| risk-score | 0~100 | 0~100 | 이미 통일 |

오케스트레이터(`index.ts`)에서 기존의 비율 환산 로직(`score / 30`, `score / 65` 등)을 제거하고, 각 모듈의 `normalizedScore` 필드를 직접 사용한다:

```
total = Σ(모듈.normalizedScore × 티어가중치) - (risk.normalizedScore × 리스크가중치)
```

### 2.4 UI — ScoreReasonPopover

기존 `UnifiedAnalysisSection.tsx`의 점수 영역 클릭 시 팝오버 표시:

- 영역별 정규화 점수 헤더 (예: "추세 72/100")
- 충족 조건: ✅ + label + points + detail
- 미충족 조건: ❌ + label + detail (왜 미충족인지)
- 리스크 감점: ⚠️ + label + points(음수) + detail
- 모바일: 바텀시트, 데스크톱: 팝오버

## 3. 접근법 C — 조건 기반 체크리스트 (신규 페이지)

### 3.1 12개 판정 조건

#### 추세 (3개)
1. **이동평균 정배열**: 5일 > 20일 > 60일선 → detail: "5일 {값} > 20일 {값} > 60일 {값}"
2. **RSI 매수구간**: 30 ≤ RSI ≤ 50 → detail: "RSI {값}"
3. **MACD/골든크로스**: 최근 3일 내 발생 → detail: "MACD 골든크로스 ({날짜})" 또는 "골든크로스 ({날짜})"

#### 수급 (3개)
4. **외국인 순매수**: 당일 > 0 → detail: "+{수량}주 ({일}일 연속)"
5. **기관 순매수**: 당일 > 0 → detail: "+{수량}주 (전환 첫날)" 등
6. **거래량 활성**: 20일 평균 대비 1.5배+ → detail: "{값}주 (평균 대비 {배}배)"

#### 밸류 (3개)
7. **PER 적정**: Forward PER < 15 또는 Trailing PER < 12 → detail: "Forward PER {값}" 또는 "PER {값}"
8. **목표주가 괴리**: 상승여력 15%+ → detail: "목표 {값} vs 현재 {값} (+{%}%)"
9. **ROE 양호**: ROE > 10% → detail: "ROE {%}%"

#### 리스크 (3개, 역방향 — "위험 없음"이 충족)
10. **과매수 없음**: RSI < 70 → detail: "RSI {값} ✅"
11. **급등 없음**: 5일 등락률 < 15% → detail: "5일 +{%}% ✅"
12. **스마트머니 이탈 없음**: 외국인·기관 동반매도 아님 → detail 상황별

### 3.2 조건 토글

- 페이지 상단 필터 패널: 12개 조건 각각 토글 스위치
- 카테고리 단위 일괄 토글 (추세/수급/밸류/리스크)
- OFF된 조건은 판정 분모에서 제외
- 설정값 `localStorage` 저장 (키: `checklist-conditions`)

### 3.3 등급 판정 (비율 기반)

| 충족 비율 | 등급 | 라벨 | 색상 |
|-----------|------|------|------|
| 80%+ | A | 적극매수 | green |
| 60~79% | B | 매수 고려 | blue |
| 40~59% | C | 관망 | orange |
| 0~39% | D | 주의 | red |

- 시가총액 티어별 가중치 적용 **없음** — 모든 종목 동일 기준

### 3.5 데이터 재사용

- 기존 `technical-score.ts`의 유틸 함수(`calcRSI`, `calcSMA`, `calcEMA` 등) import
- 데이터 조회는 기존 오케스트레이터 패턴 재사용
- API: 기존 `/api/v1/stock-ranking` 엔드포인트에 `mode=checklist` 파라미터 추가

## 5. 에러 처리

- 데이터 부족 시 (일별 가격 < 20일): 해당 조건을 "해당없음(N/A)"으로 표시, 분모에서 제외
- API 타임아웃: 기존 패턴과 동일 (에러 토스트 + 재시도 버튼)
- localStorage 접근 불가: 기본값(전체 ON) 사용

---

## 6. 테스트 계획

- 각 스코어 모듈: 기존 테스트에 `reasons` 필드 검증 추가
- 체크리스트 조건 판정: 12개 조건별 edge case 단위 테스트
- 정규화: 경계값(0, 만점, 음수) 테스트
- UI: 팝오버 열림/닫힘, 조건 토글 상태 유지 수동 확인
